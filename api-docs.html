ChatBot API

Django REST Framework API powering a multilingual RAG-only chatbot. Accepts data from Word documents and external APIs, processes them through semantic embedding and chunking, and answers questions using retrieval-augmented generation with dialect-aware responses.

Quick links

  • OpenAPI schema: GET /api/schema/
  • Swagger UI: /api/docs/
  • ReDoc: /api/redoc/

Conventions

  • Base URL: http://localhost:8000 in dev. All routes below are relative to /api/v1/.
  • Format: JSON for both requests and responses, except file uploads which use multipart/form-data.
  • Auth: Bearer JWT in the Authorization header for protected endpoints.
  • Pagination: list endpoints accept ?page=<n>&page_size=<n>; default page size is 20.
  • Filtering / search / ordering: list endpoints accept ?search=<q>, ?ordering=<field> (prefix - for descending), and resource-specific filter fields documented per endpoint.

Paginated response shape

{
  "count": 123,
  "next":  "http://…?page=2",
  "previous": null,
  "results": [ … ]
}

Auth header

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5…

Content

A pure RAG system accepts documents and external API content. All data is processed uniformly through chunking and semantic embedding.

Documents — /api/v1/documents/

Reference documents the chatbot can search. On upload the file is parsed → split into overlapping chunks → embedded → persisted as DocumentChunk rows. Once processing_status="completed", the chunks are part of the next chat retrieval.

Supported formats: .docx, .txt, .md.

Chat

Chat — POST /api/v1/chat/ public

Answer a user's question using RAG (Retrieval-Augmented Generation). The pipeline searches active document chunks by semantic similarity and generates context-aware responses in the user's language and dialect.

Pipeline

  1. Detect language and dialect — identifies Arabic dialects (Iraqi, Egyptian, Saudi, Emirati, Syrian, Moroccan) or other languages.
  2. Semantic search — retrieves top-k document chunks above RAG_SIMILARITY_THRESHOLD (default 0.45) using cosine similarity.
  3. Context-aware generation — passes chunks + conversation history + dialect-specific instructions to an LLM to generate a natural, kind, and flexible response.
  4. No information fallback — if no chunks match, returns a polite dialect-appropriate message.

Request

{
  "question": "كم سعر المنتج؟",
  "history": [
    {"role": "user",      "content": "السلام عليكم"},
    {"role": "assistant", "content": "وعليكم السلام! كيف أستطيع المساعدة؟"}
  ],
  "language": "ar"
}
FieldRequiredDefaultNotes
questionyes—≤ 2000 chars; any language
historyno[]Prior turns for context; used in prompt building
languagenoauto-detectLanguage hint; dialect detection refines it

Response — 200

{
  "answer": "السعر يعتمد على الموديل. الموديل الأساسي بـ 500 ريال...",
  "source": "rag",
  "source_id": "",
  "confident": true,
  "response_time_ms": 1250,
  "sources": [
    {
      "filename": "price_list.docx",
      "document_id": "f0e1…",
      "chunk_id": "ab12…",
      "position": 2,
      "score": 0.68
    }
  ]
}
FieldWhen / Notes
answerThe LLM's response, generated in the user's language and dialect.
sourceAlways "rag" (pure RAG system).
source_idAlways empty in RAG-only mode.
confidentfalse when no chunks matched the threshold; true otherwise.
sourcesArray of chunks that informed the answer, with similarity scores.
response_time_msTotal elapsed time from request to response (embedding + search + LLM call).

Errors

  • 400 — validation (e.g. empty question).
  • 503 — RAG stack cannot run (missing OPENAI_API_KEY, embedding/LLM provider unreachable, etc.).

WhatsApp Integration

Wraps the Meta WhatsApp Cloud API. Incoming messages are routed through the RAG chat pipeline.

Configuration (.env)

VariableNotes
WHATSAPP_ACCESS_TOKENLong-lived Meta access token
WHATSAPP_PHONE_NUMBER_IDPhone-number id to send from
WHATSAPP_VERIFY_TOKENShared secret you also enter in the Meta console

Webhook callback URL to register with Meta:

https://<your-domain>/api/v1/whatsapp/webhook/
GET/api/v1/whatsapp/webhook/public

Meta verification handshake. Echoes hub.challenge when hub.mode=subscribe and hub.verify_token matches WHATSAPP_VERIFY_TOKEN. Returns 403 on mismatch.

POST/api/v1/whatsapp/webhook/public

Receives incoming events. The view:

  1. Parses the message out of the Meta envelope.
  2. get_or_create the WhatsAppUser + active 24-h WhatsAppSession.
  3. Logs the incoming WhatsAppMessage.
  4. Routes the text to the RAG chat pipeline with optional slash-commands: /help (welcome) and /lang <code> (set language).
  5. Sends the reply via the Meta Graph API and logs the outgoing message.

Always returns 200 OK to Meta, even on internal failure (per Meta's guidance — failures are logged server-side).

POST/api/v1/whatsapp/send/auth

Send a free-form text message to a WhatsApp user.

// request
{
  "to_number": "201234567890",   // E.164, no leading +
  "message":   "تم تأكيد حجزك."
}

// response 200
{"message_id": "wamid.HBgL…", "status": "sent"}

Errors

  • 400 — missing fields.
  • 502 — Meta API rejected the request (bad token, missing phone number id, recipient outside 24-hour session window).
