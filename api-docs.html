ChatBot API

Django REST Framework API powering a knowledge-grounded chatbot, with a modern Q&A bank → RAG → fallback pipeline and a WhatsApp Cloud API integration.

Quick links

  • OpenAPI schema: GET /api/schema/
  • Swagger UI: /api/docs/
  • ReDoc: /api/redoc/

Conventions

  • Base URL: http://localhost:8000 in dev. All routes below are relative to /api/v1/.
  • Format: JSON for both requests and responses, except file uploads which use multipart/form-data.
  • Auth: Bearer JWT in the Authorization header for protected endpoints.
  • Pagination: list endpoints accept ?page=<n>&page_size=<n>; default page size is 20.
  • Filtering / search / ordering: list endpoints accept ?search=<q>, ?ordering=<field> (prefix - for descending), and resource-specific filter fields documented per endpoint.

Paginated response shape

{
  "count": 123,
  "next":  "http://…?page=2",
  "previous": null,
  "results": [ … ]
}

Auth header

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5…

Authentication

JWT via djangorestframework-simplejwt. Access tokens live for 1 hour, refresh tokens for 7 days; refresh rotation + blacklist are enabled.

Knowledge base

Two resources: a Q&A bank (the curated answers your bot should prefer) and a self-referential question tree (for guided menu-style flows). Q&A rows are auto-embedded on save by a post-save signal; failures to embed are non-fatal and can be backfilled with python manage.py embed_qa.

Documents

Documents — /api/v1/documents/

Holds reference documents the chatbot can search. On upload the file is parsed → split into overlapping chunks → embedded → persisted as DocumentChunk rows. Once processing_status="completed", the chunks are part of the next chat retrieval.

Supported formats: .docx, .txt, .md.

MethodPathNotes
GET/documents/List
POST/documents/Upload (multipart)
GET / PUT / PATCH / DELETE/documents/<id>/Standard
POST/documents/<id>/reindex/Re-chunk + re-embed

Filters: processing_status, is_active, uploaded_by. Search: filename. Ordering: created_at | updated_at | file_size.

  "is_active": true,
  "uploaded_by": 1,
  "uploaded_by_username": "alice",

Chat

Chat — POST /api/v1/chat/ public

Answer a user's question. The pipeline tries three sources in order, stopping at the first hit:

  1. Q&A bank — semantic match against QuestionAnswer above QA_BANK_THRESHOLD (default 0.82). Returns the curated answer verbatim. source="qa_bank".
  2. RAG over documents — top-k DocumentChunk above RAG_SIMILARITY_THRESHOLD (default 0.45) → context window → LLM with strict "answer only from context" prompt. source="rag".
  3. Fallback — polite "I don't have enough information" message. source="fallback", confident=false.

Request

{
  "question": "How much does the Atlas-2 Pro cost?",
  "history": [
    {"role": "user",      "content": "Hi"},
    {"role": "assistant", "content": "Hello! How can I help?"}
  ],
  "language": "en"
}
FieldRequiredDefaultNotes
questionyes—≤ 2000 chars
historyno[]Prior turns; only used by the RAG branch
languageno"ar"Hint; the model replies in the question's language regardless

Response — 200

{
  "answer": "The Atlas-2 Pro retails for €18,900.",
  "source": "rag",                 // qa_bank | rag | fallback
  "source_id": "",                 // QuestionAnswer id when source=qa_bank
  "confident": true,
  "response_time_ms": 842,
  "sources": [
    {
      "filename": "company_handbook.docx",
      "document_id": "f0e1…",
      "chunk_id": "ab12…",
      "position": 4,
      "score": 0.71
    }
  ]
}

For source="qa_bank", sources items have a different shape:

{"id": "8d1e…", "question": "…", "score": 0.91}

Errors

  • 400 — validation (e.g. empty question).
  • 503 — RAG stack cannot run (missing OPENAI_API_KEY, embedding/LLM provider unreachable, etc.).

WhatsApp Integration

Wraps the Meta WhatsApp Cloud API. Incoming messages are routed through the same chat pipeline as the HTTP API.

Configuration (.env)

VariableNotes
WHATSAPP_ACCESS_TOKENLong-lived Meta access token
WHATSAPP_PHONE_NUMBER_IDPhone-number id to send from
WHATSAPP_VERIFY_TOKENShared secret you also enter in the Meta console

Webhook callback URL to register with Meta:

https://<your-domain>/api/v1/whatsapp/webhook/
GET/api/v1/whatsapp/webhook/public

Meta verification handshake. Echoes hub.challenge when hub.mode=subscribe and hub.verify_token matches WHATSAPP_VERIFY_TOKEN. Returns 403 on mismatch.

POST/api/v1/whatsapp/webhook/public

Receives incoming events. The view:

  1. Parses the message (or status update) out of the Meta envelope.
  2. get_or_create the WhatsAppUser + active 24-h WhatsAppSession.
  3. Logs the incoming WhatsAppMessage.
  4. Routes the text to answer_question() — same pipeline as POST /chat/ — with two slash-commands handled in-process: /help (welcome) and /lang <code> (set language).
  5. Sends the reply via the Meta Graph API and logs the outgoing message.

Always returns 200 OK to Meta, even on internal failure (per Meta's guidance — failures are logged server-side).

POST/api/v1/whatsapp/send/auth

Send a free-form text message to a WhatsApp user.

// request
{
  "to_number": "201234567890",   // E.164, no leading +
  "message":   "Your booking has been confirmed."
}

// response 200
{"message_id": "wamid.HBgL…", "status": "sent"}

Errors

  • 400 — missing fields.
  • 502 — Meta API rejected the request (bad token, missing phone number id, recipient outside the 24-hour session window).
